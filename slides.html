Agents operate with Partial Knowledge. Local rationality is preserved as agents only perceive their immediate environment and the availability of global grants.

Agent Autonomy

The Tech Stack: Performance & Safety

Bun.js Runtime

Chosen for its native high-performance. In a DAI context, Bun enables sub-millisecond Perception-Action loops, allowing the simulation to scale to dozens of agents without latency drift.

Strictly Typed TypeScript

Type-safety is used to define Protocol Contracts between agents. This prevents runtime failures in the asynchronous, event-driven handshake of the system.

Communication & Fault Tolerance

Temporal & Spatial Decoupling

MQTT (Mosquitto) allows agents to interact without knowing each other's network location or availability at the exact moment of transmission.

LWT (Last Will & Testament)

The system implements Last Will patterns. If a sensor agent crashes, the Room Agent enters a Fail-Safe state (e.g., HVAC safety shutdown) to prevent environmental drift.

Trade-off: Fairness vs. Efficiency

I implemented Round-Robin scheduling. While a greedy algorithm would be more energy-efficient, Round-Robin ensures Zero-Starvation, guaranteeing every room eventually reaches its comfort setpoint.

Quantitative vs. Qualitative Analysis

The Metrics (Quantitative)

Tracking Decision Latency and Message Throughput. These metrics prove the technical stability of the MQTT backbone under load.

The Insight (Qualitative)

The data reveals that under strict energy caps, Comfort Violations follow a predictable pattern. This proves the Orchestrator successfully maintains building safety over individual room comfort.

Experimental Finding: As agent density increases, the Orchestrator becomes the primary bottleneck. Future iterations suggest a Decentralized Auctioning system to further distribute coordination.

Technical Defense & Q&A

EXAM READINESS

1. Is the system truly decentralized?

Control is distributed among autonomous room agents. Coordination of shared resources is managed by a specialized Orchestrator to prevent building-wide energy surges, balancing local autonomy with global stability.

2. Handling Network Partitions?

Agents are designed with Local Persistence. If the MQTT broker drops, agents enter a "Local-Only" safety loop, holding the last safe actuator state until connectivity is restored.

3. Why PostgreSQL + MQTT?

It is a Hybrid State Model. Postgres manages static provisioning (SaaS metadata), while MQTT handles the high-frequency dynamic state. This is the industrial standard for scalable IoT Digital Twins.
